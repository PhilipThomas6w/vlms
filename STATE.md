Every sign-in threw, so `UserProvisioningService.FindOrCreateAsync` never ran and no `AppUser`/`UserRole` rows were ever created — a genuine regression versus the prior `IHttpContextAccessor` code, which failed closed (empty principal) rather than throwing. **Fix:** made principal resolution genuinely lazy instead of eager. `EntraCurrentUserContext` gained a second constructor overload taking `AuthenticationStateProvider` (rather than an already-resolved `ClaimsPrincipal`) and defers calling `AuthenticationStatePrincipalResolver.Resolve` until inside its existing `Lazy<...>`, forced only by a first read of `UserId`/`HasRole` (`src/Vlms.Infrastructure/Security/EntraCurrentUserContext.cs`); the original `ClaimsPrincipal` constructor is unchanged and still used by existing tests. `Program.cs`'s factory now just passes the `AuthenticationStateProvider` through, with no `Resolve` call of its own. `UserProvisioningService` never reads `ICurrentUserContext.UserId`/`HasRole`, so the OIDC path constructs `EntraCurrentUserContext` but never forces resolution — no throw, regardless of DI scope. 6 new tests added to `AuthenticationStatePrincipalResolverTests.cs`: a `ThrowingUntilPrimedAuthenticationStateProvider` fake reproduces the real `ServerAuthenticationStateProvider`'s unprimed-throw behaviour (verified it actually throws, so it's not a no-op stand-in);
