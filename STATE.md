  - **Checker-FAIL round-trip fix (regression in the above):** an adversarial (Opus) checker review of d2adf82 found the interactive-render-mode fix itself broke sign-in: `Program.cs`'s `ICurrentUserContext` DI factory called `AuthenticationStatePrincipalResolver.Resolve(authenticationStateProvider)` *eagerly*, at construction time. That same factory also runs during the OIDC `OnTokenValidated` callback (`UserProvisioningService` → `VlmsDbContext` → `ICurrentUserContext`), which is not a rendered Razor component's DI scope — the real ASP.NET Core `ServerAuthenticationStateProvider` throws `InvalidOperationException` from `GetAuthenticationStateAsync()` there ("Do not call GetAuthenticationStateAsync outside of the DI scope for a Razor component..."), because no component has called `SetAuthenticationState` yet. Every sign-in threw, so `UserProvisioningService.FindOrCreateAsync` never ran and no `AppUser`/`UserRole` rows were ever created — a genuine regression versus the prior `IHttpContextAccessor` code, which failed closed (empty principal) rather than throwing. **Fix:** made principal resolution genuinely lazy instead of eager. `EntraCurrentUserContext` gained a second constructor overload taking `AuthenticationStateProvider` (rather than an already-resolved `ClaimsPrincipal`) and defers calling `AuthenticationStatePrincipalResolver.Resolve` until inside its existing `Lazy<...>`, forced only by a first read of `UserId`/`HasRole` (`src/Vlms.Infrastructure/Security/EntraCurrentUserContext.cs`); the original `ClaimsPrincipal` constructor is unchanged and still used by existing tests. `Program.cs`'s factory now just passes the `AuthenticationStateProvider` through, with no `Resolve` call of its own. `UserProvisioningService` never reads `ICurrentUserContext.UserId`/`HasRole`, so the OIDC path constructs `EntraCurrentUserContext` but never forces resolution — no throw, regardless of DI scope. 6 new tests added to `AuthenticationStatePrincipalResolverTests.cs`: a `ThrowingUntilPrimedAuthenticationStateProvider` fake reproduces the real `ServerAuthenticationStateProvider`'s unprimed-throw behaviour (verified it actually throws, so it's not a no-op stand-in); a test proving the *old* eager-resolve-then-construct shape throws with it (documents the regression); a test proving the new `AuthenticationStateProvider` constructor never throws at construction against the unprimed fake; a test proving reads of `UserId`/`HasRole` still surface the throw when genuinely unprimed (laziness isn't silently swallowing); a test proving resolution works correctly once primed; and — the property that actually matters — a test exercising the real `UserProvisioningService` → `VlmsDbContext` → `ICurrentUserContext` chain against the unprimed throwing fake, asserting `FindOrCreateAsync` completes and persists the new `AppUser` without ever forcing the throw. `openwiki/authentication-authorization.md` and `openwiki/web.md` corrected: no longer claim resolution is unconditionally safe ("by the time anything resolves ICurrentUserContext..."/"There is no longer a known gap"); now distinguish "safe for component-scope consumers because the framework has already captured `AuthenticationState`" from "safe for the OIDC path because resolution is deferred until first read, which that path never triggers" — and document the regression and fix. — commit 28b4b83, `pwsh -File build/verify.ps1` green (build -warnaserror clean, 40/40 tests incl. 6 new, secrets scan clean, `dotnet ef migrations has-pending-model-changes` confirms no drift — no model changes in this fix)
