# Safeguarding & consent: management UI + ConsentExpiryJob

Design source: `docs/design/low-level-design.md` "ConsentExpiryJob"/"NotificationService", `docs/design/data-design.md` (`ConsentRecord`/`ConsentSensitiveDetails`/`DbsCheck`), `docs/requirements/functional.md` FR-001/FR-002/FR-003, `docs/adr/0003-scheduled-jobs-webjobs.md`, `docs/adr/0004-sensitive-data-access-control.md`. Read [access-control.md](access-control.md) first — this page builds directly on its query-filter/audit mechanism, doesn't change it.

## Management services (`src/Vlms.Infrastructure/Safeguarding/`)

`ConsentRecordService` and `DbsCheckService` — same shape and conventions as `GuardianLinkService`/`StudentRegistrationService` (see [guardian-links.md](guardian-links.md)): role-checked inside the service itself (defense in depth), Admin or SafeguardingOfficer only. This matches data-design.md's own wording for who may act on these entities, and is a hard constraint (CLAUDE.md Project Law): the Approver role is curriculum-only and has zero access here, and there is no Parent self-service path yet (the Parent dashboard/self-submission flow is a separate `STATE.md` Next item, not built) — these services are the Admin/SafeguardingOfficer-operated equivalent of entering what a parent has already told the programme in writing.

- **`ConsentRecordService.RecordAsync(...)`** — creates a `ConsentRecord` (Status starts Pending) plus its 1:1 `ConsentSensitiveDetails` row in a single `SaveChangesAsync` call. Both rows' ids are computed up front (this codebase's ids are application-assigned, not database-identity — see `VlmsDbContext.OnModelCreating`), so both can be added to the change tracker before any database round-trip — atomic by construction, not by an explicit `BeginTransactionAsync`, avoiding the non-atomicity bug class a prior increment's checker review found in `StudentRegistrationService` (commit d2f9c63).
- **`ConsentRecordService.DecideAsync(consentRecordId, approve)`** — the only path that sets `Status` to Approved/Rejected, recording `ApprovedByUserId` as the deciding caller. Mirrors `LessonProposalService`'s propose/decide shape.
- **`DbsCheckService.RecordAsync(...)`** — creates a `DbsCheck`, after checking the target `AppUser` actually holds `Role.Teacher` (data-design.md: "DBS check tracking for teachers").
- **`DbsCheckService.UpdateStatusAsync(dbsCheckId, newStatus)`** — updates a check's `DbsCheckStatus` (e.g. Pending → Clear/Flagged once a result arrives).

Neither service ever calls `IgnoreQueryFilters()` or uses raw SQL — all reads/writes go through the normal `VlmsDbContext`, so the existing whole-entity query filter and `SensitiveDataAuditInterceptor` read-audit apply exactly as everywhere else.

## Pages (`src/Vlms.Web/Components/Pages/Safeguarding/`)

`ConsentRecords.razor` (`/safeguarding/consent`) and `DbsChecks.razor` (`/safeguarding/dbs`), both gated by a new `[Authorize(Policy = "RequireAdminOrSafeguardingOfficer")]` policy (`AnyRoleRequirement(Role.Admin, Role.SafeguardingOfficer)` — see [authentication-authorization.md](authentication-authorization.md)'s "Multi-role policy" section). Same `EditForm`/`InputSelect` conventions as `GuardianLinks.razor`/`RegisterStudent.razor`. Reading `Db.ConsentSensitiveDetails`/`Db.DbsChecks` directly from these pages (for the "existing records" table) only resolves rows because the page is role-gated the same way the query filter requires — and every materialized row writes a `SensitiveDataAccessLog` entry via the existing interceptor, exercising the audit trail for real through this UI (not just in tests).

## `ConsentExpiryJob` (`src/Vlms.Infrastructure/Safeguarding/ConsentExpiryJob.cs`)

The daily sweep named in low-level-design.md. Fully unit-tested independently of any host (`tests/Vlms.Tests/Infrastructure/ConsentExpiryJobTests.cs`) — `RunAsync()` role-checks (Admin/SafeguardingOfficer), then computes three things and returns a `ConsentExpiryJobResult`:

1. **Consent flags** (`SweepConsentAsync`) — for each *Active* student, finds their most recent Approved `ConsentRecord` (by `ExpiryDate`, computed in memory after materializing rather than via EF `GroupBy`/`OrderBy`/`First` translation — deliberately, since that shape isn't reliably supported across providers). `IsExpired = true` if none exists at all, or the latest one's `ExpiryDate < today` — the exact same condition `CompletionService.RequireActiveConsentAsync` already blocks lesson completion on, so a flag here is exactly the population currently blocked. Otherwise flagged (not expired) if the expiry falls within the warning window.
2. **DBS flags** (`SweepDbsAsync`) — mirrors the consent shape, but per `AppUser` holding `Role.Teacher` (via `UserRole`) and per Clear `DbsCheck`. A teacher who's never had a Clear check (never checked, or their latest result is Flagged/Pending) is `IsExpired = true` — there's no separate "block" mechanism for missing/invalid DBS the way FR-003 gives consent, so this job's escalation output is the only structural signal for it.
3. **At-risk students** (`FlagAtRiskStudentsAsync`) — functional.md's "no lesson completions within 8 weeks" (56 days), Active students only. `LastActivityAt` is the latest non-reversed `StudentLessonCompletion.CompletedAt`, or `EnrolmentDate` if there is none — so a newly enrolled student isn't flagged before they've had a fair chance.

### Judgement calls this class documents and this page repeats for visibility

- **"At-risk flagging at 8 weeks" is disengagement, not consent expiry.** low-level-design.md's `ConsentExpiryJob` paragraph explicitly bundles two different concerns into one job — daily consent/DBS expiry sweep *and* "at-risk/disengaged student flagging: no lesson completion within 8 weeks" — even though the second is about engagement, not safeguarding-document lapse. That bundling is the existing design decision (STATE.md's item wording mirrors it exactly); this codebase followed it rather than inventing a split.
- **The 28-day consent/DBS warning window is a documented default, not a discovered number.** low-level-design.md calls it "configurable" but never names a figure, and no other doc does either. `ConsentExpiryJob.DefaultExpiryWarningWindowDays` (28) is a build-time default; the constructor's `expiryWarningWindowDays` parameter is what makes it genuinely configurable, per the ADR's own wording.
- **"Escalation" was an elevated-severity structured log entry only, until `NotificationService`/`ConsentExpiryNotifier` landed (see [notifications.md](notifications.md)).** `ConsentExpiryJob.LogFlags` still logs expired/missing consent and DBS at `LogError` (matching low-level-design.md's "escalates to Admin/Safeguarding Officer" wording for that specific case) and everything else — approaching-threshold consent/DBS, at-risk students — at `LogWarning`; ADR-0003 names Application Insights alerting as this WebJob's monitoring mechanism, which is exactly what a `LogError` from a scheduled process feeds. `Vlms.Jobs/Program.cs` now calls `ConsentExpiryNotifier.NotifyAsync(result)` immediately after `RunAsync()` returns — **alongside** this logging, not instead of it (removing the logging would reduce the WebJob's general observability for no reason) — translating the two `IsExpired = true` flag categories (no valid consent, no valid DBS) into a real digest email to every Admin/SafeguardingOfficer `AppUser`, with retry/backoff and a final escalation-visible log entry if every attempt fails. Approaching-threshold flags and at-risk students remain log-only (see notifications.md's `ConsentExpiryNotifier` scope decisions).
- **"No consent record at all" is treated as expired, not as a separate case** — this mirrors `CompletionService`'s own blocking condition exactly (see above), so the two are guaranteed to describe the same population without duplicating the rule.
- **Scoped to Active students/all-time teachers** — a Graduated/Inactive student needs no ongoing monitoring for either consent expiry or disengagement; there's no equivalent "inactive teacher" concept in this schema, so the DBS sweep considers every `AppUser` ever assigned `Role.Teacher`.

## `SystemCurrentUserContext` (`src/Vlms.Infrastructure/Security/SystemCurrentUserContext.cs`)

The `ICurrentUserContext` the WebJob host uses — there is no signed-in human caller for a scheduled job. Grants exactly the two roles the sensitive-data query filter checks for (`Role.Admin`, `Role.SafeguardingOfficer`) and nothing else — not a general bypass, just what the job legitimately needs to read `DbsCheck` through the normal filtered/audited path. `UserId` is null, which both `ICurrentUserContext.UserId`'s own doc comment and `SensitiveDataAccessLog.UserId` already model as a valid, resolvable-caller-absent case. Never wired into `Vlms.Web` — only used by the `Vlms.Jobs` host below.

## `Vlms.Jobs` — the WebJob host (`src/Vlms.Jobs/`)

The first WebJob in this solution (`docs/adr/0003-scheduled-jobs-webjobs.md`). A plain console app (`Host.CreateApplicationBuilder`, not the WebJobs SDK — that SDK's Azure Storage-backed dashboard/queue-trigger machinery is more than a single daily Triggered WebJob needs, and ADR-0003 explicitly favours fewest moving parts). `Program.cs` wires a `VlmsDbContext` (SQL Server), `SystemCurrentUserContext`, and calls `ConsentExpiryJob.RunAsync()` once; it now also wires `EmailClient`/`AzureCommunicationEmailSender`/`NotificationService`/`ConsentExpiryNotifier` and calls `ConsentExpiryNotifier.NotifyAsync(result)` straight after — see [notifications.md](notifications.md) — before exiting (Triggered WebJob semantics). `settings.job` (an NCRONTAB expression, confirmed against [Microsoft Learn's WebJobs documentation](https://learn.microsoft.com/en-us/azure/app-service/webjobs-create)) schedules it daily at 03:00 UTC.

Not deployed by anything in this repository (no live Azure environment exists yet — the same "structurally correct, not yet wired" status `AzureBlobStorage`/`EntraCurrentUserContext` had before their dependencies existed, and now also `AzureCommunicationEmailSender`'s: `CommunicationServices:ConnectionString`/`SenderAddress` in `appsettings.json` are placeholders). In Azure, this project's publish output would be deployed into `Vlms.Web`'s App Service under `App_Data/jobs/triggered/ConsentExpiryJob/`, co-located with the same App Service plan per the ADR; Application Settings (including the connection string) are injected as environment variables the same way they are for the web app itself.

## Tests

`tests/Vlms.Tests/Infrastructure/ConsentRecordServiceTests.cs`, `DbsCheckServiceTests.cs`, `ConsentExpiryJobTests.cs` — same SQLite-in-memory-via-DI pattern as `SensitiveDataAccessControlTests` (required here too, since these services/the job all touch `DbsCheck`/`ConsentSensitiveDetails`). `ListLogger<T>` (`tests/Vlms.Tests/Infrastructure/ListLogger.cs`) is a hand-rolled `ILogger<T>` test double, same spirit as `FakeCurrentUserContext`/`InMemoryBlobStorage` — records level+message so tests can assert on escalation severity without a mocking library. `ConsentExpiryNotifierTests.cs` covers the notification-translation layer — see [notifications.md](notifications.md).
